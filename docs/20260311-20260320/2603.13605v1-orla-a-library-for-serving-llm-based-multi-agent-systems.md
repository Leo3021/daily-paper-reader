## 摘要
我们介绍了 Orla，这是一个用于构建和运行基于大语言模型（LLM）的智能体系统的库。现代智能体应用由结合了多个 LLM 推理步骤、工具调用和异构基础设施的工作流组成。目前，开发人员通常通过手动将编排代码与 LLM 服务引擎和工具执行逻辑相结合来构建这些系统。Orla 提供了一种通用的抽象，将请求执行与工作流级策略分离。它作为现有 LLM 推理引擎之上的服务层：开发人员定义由阶段（stages）组成的工作流，而 Orla 负责管理这些阶段如何在不同模型和后端之间进行映射、执行和协调。它通过三种机制提供智能体级控制：阶段映射器（stage mapper），负责将每个阶段分配给合适的模型和后端；工作流编排器（workflow orchestrator），负责调度阶段并管理其资源和上下文；以及内存管理器（memory manager），负责跨工作流边界管理推理状态（如 KV 缓存）。我们通过一个展示了其多项能力的客户支持工作流来演示 Orla。我们在两个数据集上对 Orla 进行了评估，结果表明，与单模型 vLLM 基准相比，阶段映射改善了延迟和成本，而工作流级缓存管理减少了首字延迟（time-to-first-token）。

## Abstract
We introduce Orla, a library for constructing and running LLM-based agentic systems. Modern agentic applications consist of workflows that combine multiple LLM inference steps, tool calls, and heterogeneous infrastructure. Today, developers typically build these systems by manually composing orchestration code with LLM serving engines and tool execution logic. Orla provides a general abstraction that separates request execution from workflow-level policy. It acts as a serving layer above existing LLM inference engines: developers define workflows composed of stages, while Orla manages how those stages are mapped, executed, and coordinated across models and backends. It provides agent-level control through three mechanisms: a stage mapper, which assigns each stage to an appropriate model and backend; a workflow orchestrator, which schedules stages and manages their resources and context; and a memory manager, which manages inference state such as the KV cache across workflow boundaries. We demonstrate Orla with a customer support workflow that exercises many of its capabilities. We evaluate Orla on two datasets, showing that stage mapping improves latency and cost compared to a single-model vLLM baseline, while workflow-level cache management reduces time-to-first-token.

*   **核心目标**：提出 Orla 库，通过在现有推理引擎之上构建一个通用的服务层，实现请求执行与工作流策略的分离，从而优化多智能体系统的性能和成本。

#### 2. 论文提出的方法论
Orla 的核心思想是将智能体任务抽象为由“阶段（Stage）”组成的有向无环图（DAG），并通过以下三个关键组件进行管理：
*   **阶段映射器 (Stage Mapper)**：负责将工作流中的每个阶段分配给最合适的模型和后端。支持显式映射（开发者指定）和动态路由（如 `OneBitStageMapper`，利用轻量级模型判断任务复杂度并分流至大/小模型）。
*   **工作流编排器 (Workflow Orchestrator)**：
    *   **调度**：采用两级调度机制，首先选择要服务的阶段队列（支持 FCFS、优先级调度等），然后在阶段内选择具体请求。
    *   **资源与上下文管理**：负责在阶段间传递输出（如 JSON 或工具结果），并利用构建器函数（Builder functions）动态生成后续阶段的 Prompt。
*   **内存管理器 (Memory Manager)**：在工作流层面管理 KV 缓存。它能识别阶段间的依赖关系，决定何时保留缓存以避免重复 Prefill（如连续阶段使用同一后端且上下文前缀重合），以及何时在内存压力下释放缓存。

#### 7. 优点
*   **解耦设计**：将复杂的智能体逻辑与底层的推理引擎解耦，使 Orla 能够作为“即插即用”的库运行在 vLLM、SGLang 或 Ollama 之上。
*   **研究友好**：提供了模拟后端（Simulated Backend），允许研究人员在没有 GPU 的情况下研究调度和协调算法。
*   **多维度优化**：不仅关注吞吐量，还通过工作流感知同时优化了成本、延迟和资源利用率。

#### 8. 不足与局限
*   **动态路由策略较简单**：目前使用的 `OneBitStageMapper` 仅依赖简单的分类，对于极度复杂的多智能体交互，可能需要更精细的路由算法。

*   **后端依赖**：虽然支持多种后端，但某些高级功能（如精细的 KV 缓存控制 API）仍依赖于后端引擎（如 SGLang）的开放程度。
