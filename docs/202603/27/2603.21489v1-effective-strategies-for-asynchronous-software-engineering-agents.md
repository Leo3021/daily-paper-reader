## 摘要
AI 智能体在处理孤立的软件工程（SWE）任务（如解决 Github 上的问题）方面能力日益增强。然而，涉及多个相互依赖子任务的长程任务在准确性和及时完成方面仍面临挑战。及时解决这些长程任务的一种自然方法是异步多智能体协作，即多个智能体同时处理任务的不同部分。但多智能体系统的有效应用已被证明异常困难：多个智能体的并发编辑会相互干扰，依赖关系难以同步，且将部分进展合并为连贯的整体具有挑战性。另一方面，人类开发者长期以来依靠成熟的协作基础设施来管理大型软件项目中的这些挑战。受这些协作原语的启发，我们引入了集中式异步隔离委托（CAID），这是一种基于三个核心 SWE 原语的结构化多智能体协调范式：集中式任务委托、异步执行和隔离工作区。CAID 通过中央管理器构建具有依赖感知能力的任务计划，在隔离工作区中并发执行子任务，并通过基于可执行测试验证的结构化集成来整合进展。在实证评估中，我们发现 CAID 在论文复现任务（PaperBench）上比单智能体基准的准确率绝对提升了 26.7%，在 Python 库开发任务（Commit0）上提升了 14.3%。通过系统分析，我们发现“分支与合并”是多智能体协作的核心协调机制，而 git worktree、git commit 和 git merge 等 SWE 原语使其能够以可靠且可执行的方式实现。

## Abstract
AI agents have become increasingly capable at isolated software engineering (SWE) tasks such as resolving issues on Github. Yet long-horizon tasks involving multiple interdependent subtasks still pose challenges both with respect to accuracy, and with respect to timely completion. A natural approach to solving these long-horizon tasks in a timely manner is asynchronous multi-agent collaboration, where multiple agents work on different parts of the task at the same time. But effective application of multi-agent systems has proven surprisingly difficult: concurrent edits by multiple agents interfere with each other, dependencies are difficult to synchronize, and combining partial progress into a coherent whole is challenging. On the other hand, human developers have long relied on mature collaboration infrastructure to manage these challenges in large software projects. Inspired by these collaboration primitives, we introduce Centralized Asynchronous Isolated Delegation (CAID), a structured multi-agent coordination paradigm grounded in three core SWE primitives: centralized task delegation, asynchronous execution, and isolated workspaces. CAID constructs dependency-aware task plans through a central manager, executes subtasks concurrently in isolated workspaces, and consolidates progress via structured integration with executable test-based verification. In empirical evaluation, we find that CAID improves accuracy over single-agent baselines by 26.7% absolute on paper reproduction tasks (PaperBench) and 14.3% on Python library development tasks (Commit0). Through systematic analysis, we find that branch-and-merge is a central coordination mechanism for multi-agent collaboration, and that SWE primitives such as git worktree, git commit, and git merge enable it to be realized in a reliable and executable manner.

#### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）的发展，AI 智能体在处理孤立的软件工程（SWE）任务（如修复单个 Bug）上表现出色。然而，面对**长周期、多任务相互依赖**的复杂项目（如从零开发一个库或复现一篇科研论文）时，现有系统面临两大瓶颈：
*   **单智能体局限性：** 任务跨度过长导致上下文窗口爆炸、推理疲劳，且耗时极长。
*   **多智能体协作难题：** 简单的多智能体协作常因并发编辑冲突（相互覆盖代码）、依赖同步困难以及缺乏有效的进度整合机制，导致“1+1<2”的尴尬局面。
**研究动机：** 借鉴人类开发者成熟的协作模式（如 Git 分支管理），探索如何通过结构化的软件工程原语来协调多个智能体进行高效、无冲突的异步协作。

#### 2. 方法论：核心思想与关键技术
论文提出了 **CAID (Centralized Asynchronous Isolated Delegation)** 架构，其核心思想是将“分支与合并”机制引入多智能体系统。

*   **核心组件：**
    *   **中心化任务委派 (Manager)：** 管理器构建任务依赖图 $G=(V, E)$，将大任务分解为可并行化的子任务组。
    *   **异步执行 (Engineers)：** 多个工程师智能体作为独立协程运行，互不阻塞。
    *   **隔离工作区 (Isolation)：** 利用 `git worktree` 为每个工程师创建物理隔离的独立工作目录，确保并发修改不会产生静默覆盖。
*   **工作流程：**
    1.  **规划：** Manager 分析代码库依赖，确定哪些文件/函数可以并行开发。
    2.  **委派：** 通过结构化 JSON 协议向工程师下达指令。
    3.  **实现与自验证：** 工程师在隔离分支中编写代码，并必须运行可执行测试进行自检。
    4.  **集成：** 工程师提交 `git commit`，Manager 通过 `git merge` 将其合并回主分支。若发生冲突，由产生冲突的工程师负责拉取主分支并本地解决。
    5.  **动态更新：** 每次合并后，Manager 动态更新依赖状态并指派下一批任务。

#### 6. 主要结论与发现
*   **性能飞跃：** CAID 在 PaperBench 上比单智能体准确率提升了 **26.7%**，在 Commit0 上提升了 **14.3%**。
*   **迭代边际效应：** 单纯增加单智能体的迭代次数（从 100 增加到 200）几乎不能提升性能，甚至可能因上下文污染导致性能下降。
*   **隔离的重要性：** 物理隔离（`git worktree`）远优于基于指令的软隔离，尤其在任务边界模糊的论文复现任务中。
*   **并行度的“甜点位”：** 增加智能体数量并非越多越好（如 8 个智能体可能不如 4 个），过度的并行会增加 Manager 的协调负担和集成冲突。

#### 7. 优点与亮点
*   **架构稳健：** 将成熟的软件工程实践（Git 流程）直接转化为智能体协作协议，解决了多智能体并发修改的物理冲突问题。
*   **结构化通信：** 使用 JSON 协议而非纯自然语言对话，显著降低了智能体间的理解偏差。
*   **自愈能力：** 引入了工程师负责制（谁合并谁解决冲突）和强制自验证（测试驱动），保证了集成后的代码质量。

#### 8. 不足与局限
*   **成本与耗时：** 尽管成功率提升，但多智能体系统的 API 成本显著高于单智能体，且由于集成和测试环节是串行的，实际墙钟时间（Wall-clock time）并未因并行而大幅缩短。
*   **Manager 瓶颈：** 系统的成败高度依赖于 Manager 的任务分解能力。如果 Manager 规划错误（如遗漏关键依赖），整个团队都会失败。
*   **领域局限：** 目前仅在软件工程领域验证，对于缺乏明确“测试用例”或“版本控制”概念的非代码长程任务（如创意写作、法律分析），该模式的迁移性有待观察。
