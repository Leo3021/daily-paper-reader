# Turn: A Language for Agentic Computation

Abstract 

We present Turn , a compiled, actor-based programming language—statically typed for schema inference, dynamically typed at the value level—for agentic software: programs that reason and act autonomously by delegating inference to large language models (LLMs). Existing approaches augment general-purpose languages with frameworks, encoding critical invariants (bounded context, typed inference output, credential isolation, durable state) as application-level conventions rather than language guarantees. Turn introduces five language-level constructs that address this gap. Cognitive Type Safety makes LLM inference a typed primitive: the compiler generates a JSON Schema from a struct definition and the VM validates model output before binding. The confidence operator enables deterministic control flow gated on model certainty. Turn’s actor-based process model , derived from Erlang, gives each agent an isolated context window, persistent memory, and mailbox. A capability-based identity system returns opaque, unforgeable handles from the VM host, ensuring raw credentials never enter agent memory. Finally, compile-time schema absorption (use schema::<protocol> )synthesizes typed API bindings from external specifications at compile time; the openapi adapter is shipped with graphql , fhir , and mcp in active development. We describe the language design, type rules, schema semantics, and a Rust-based bytecode VM, and evaluate Turn against representative agentic workloads. Turn is open source at 

https://github.com/ekizito96/Turn .

Keywords: programming languages, agentic software, LLM integration, actor model, type safety, capability-based security. 

# 1 Introduction 

The emergence of large language models (LLMs) as general-purpose reasoning engines has produced a new class of software: agentic programs that observe their environment, reason over unstructured data, invoke external tools, and act on their conclusions. Unlike traditional software, where all behavior is deterministic, agentic programs delegate critical decision-making to stochastic models whose outputs are structurally unpredictable. Building these systems with existing lan-guages and frameworks reveals a fundamental impedance mismatch. The core abstractions of agentic computation—bounded context win-dows, typed inference results, durable execu-tion state, process-isolated memory, and cre-dential security—have no direct representation in languages like Python, TypeScript, or Rust. Developers encode them as library conventions, leading to five recurring failure modes: 1. Unbounded context. Context is man-aged as a plain list of messages. When it exceeds the model’s token limit, it over-flows silently or truncates from the front, causing the model to lose critical informa-tion. 2. Untyped inference output. The model returns a string. The program parses it as JSON with no guarantee that the schema matches. Schema violations surface as runtime key errors with no connection to the inference that produced them. 3. Fragmented state. Agent state is scat-tered across variables, databases, API re-sponses, and framework internals. There is no single, serialisable representation of what the agent knows or has done. 1

# 2 Background and Related Work 

2.1 Agent Frameworks 

The dominant approaches to agentic software are library-based: LangChain [ 8 ] chains LLM calls with tool invocations in Python; Auto-GPT [ 9 ] implements a goal-directed decompo-sition loop; CrewAI [ 10 ] organises role-based agent teams; Semantic Kernel [ 11 ] provides a plugin architecture for .NET; MetaGPT [ 22 ] as-signs agents distinct roles in a software pipeline. At the interaction pattern level, Yao et al. [20 ] introduced ReAct—interleaved reasoning traces and tool actions—which has become the standard agent loop template, and Park et al. [ 21 ] demonstrated persistent-memory multi-agent simulation. These approaches share a fundamental lim-itation: they are libraries, not languages. The invariants they attempt to enforce—context bounds, output schemas, tool permissions— exist as runtime conventions that the host lan-guage cannot statically verify. Kizito [ 16 ] pro-vides empirical evidence that autonomous sub-agent loops fail at a rate governed by stochastic accumulation ( 0.95 20 = 0 .36 ), and that reliabil-ity requires typed boundaries and deterministic repair routing rather than better prompts. 

2.2 Programming Language Theory 

Turn draws on several established traditions in programming language design. 

Actor model. The actor model, introduced by Hewitt et al. [ 1] and formalized by Agha [18 ], provides the theoretical foundation for Turn’s concurrency. Erlang [ 2] demonstrated that actor-based concurrency with “let it crash” fault tolerance scales to millions of processes. Turn adapts this model to agents where each process has not only an environment and mail-box but also a context window and persistent memory. 

Capability-based security. Dennis and Van Horn [3 ] established capability-based access control as an alternative to access control lists. Miller [ 19 ] extended this to object-capability discipline in the E pro-gramming language, demonstrating that unforgeable references can enforce security properties compositionally. Turn’s Identity 

type implements this object-capability model: capabilities are unforgeable tokens granted by the runtime, never strings stored in the program’s memory. 

Session types and structured communica-tion. The idea that communication protocols can be enforced by the type system has been explored extensively [ 4, 5 ]. Turn’s Cognitive Type Safety applies a similar principle to the “communication protocol” between a program and an LLM: the struct schema serves as a ses-sion type constraining the model’s response. 

Orthogonal persistence. The concept of transparent persistence, where the programmer writes no explicit save/load code, was explored in PS-algol [ 6] and Napier88 [ 7 ]. Turn’s suspend primitive and 

remember /recall store provide orthogonal persistence at the process level. 

2.3 Attention and Context in LLMs 

Liu et al. [ 23 ] empirically demonstrated the “Lost in the Middle” phenomenon: informa-tion placed in the middle of long contexts is recalled with approximately 50% probability, compared to 90% for primacy and 85% for re-cency positions. This U-shaped attention curve has direct implications for language design. Turn’s per-process context isolation (§6) and the context.append /context.system primi-tives give the programmer explicit control over what occupies the high-recall zones of the con-text window, rather than accumulating an un-bounded message list. Wei et al. [ 24 ] showed 3Turn: A Language for Agentic Computation Preprint 

that chain-of-thought prompting elicits multi-step reasoning, but Kizito [ 15 ] demonstrated that this reasoning degrades predictably as context entropy increases, motivating Turn’s active context management at the language level. 

2.4 Agent Identity and Security 

The security of autonomous agents has received increasing attention as agents are deployed in production environments with access to sensi-tive APIs. South et al. [ 25 ] survey the emerg-ing landscape of identity management for agen-tic AI, highlighting that existing authoriza-tion frameworks are designed for interactive humans or static servers. Sangalo [ 17 ] pro-poses the Nexus Protocol as a standardized open protocol for agent identity and connec-tion orchestration, decoupling authentication mechanics from agent logic through a central authority. Turn addresses a complementary problem at a different layer: even when the identity infrastructure is correct, the LLM it-self must never see raw credentials. Turn’s 

grant identity primitive (§7) ensures that the token is opaque and unforgeable at the lan-guage level, regardless of how the underlying identity infrastructure is implemented. 

2.5 Structured LLM Output 

OpenAI’s structured outputs API [ 12 ] and An-thropic’s tool-use protocol [ 13 ] allow request-ing JSON-conforming responses from models. These are HTTP-level features accessed via SDK calls. Turn lifts this capability into the language itself: the struct definition is the schema, the compiler generates it, and the VM enforces it. The programmer never writes or reads a JSON Schema directly. 

# 3 Language Design 

Turn is an ahead-of-time compiled, dynam-ically typed, expression-oriented language. Source programs are compiled to bytecode and executed by a stack-based virtual machine. The core abstraction is the turn : a named unit of agent behavior analogous to a function body but with implicit access to an LLM context window, persistent memory, and an actor mailbox. 

3.1 Core Syntax 

A Turn program is a sequence of statements executed within an implicit top-level turn. The six core primitives— struct (type definition), 

context (LLM input management), infer 

(typed inference), confidence (probabilistic branching), call (tool invocation), and 

remember (persistent memory)—are exercised together in the running example.

Design Rationale: Targeted Strictness. 

A common critique of hybrid systems is the irony of enforcing strict types on non-deterministic LLM output while leaving deterministic internal variables dynamically typed. Turn adopts this targeted strictness 

deliberately. Agentic glue code is highly ex-ploratory, heavily IO-bound, and structurally resembles scripting (e.g., Python, Ruby); dynamic variables maximize iteration speed. Strict static typing is applied exactly at the boundaries of maximum entropy (LLM inference) and maximum risk (API contracts). The Identity type is similarly restricted: it cannot be coerced to Str , serialized to JSON, 4Turn: A Language for Agentic Computation Preprint 

or passed to output tools, ensuring strictness only where security demands it. 

# 4 Cognitive Type Safety 

The central innovation of Turn is the treatment of LLM inference as a typed language primitive 

rather than an untyped function call. 

4.1 The infer Primitive 

Definition 1 (Inference Expression) . An in-ference expression has the form: 

let x = infer T { e }

where T is a struct type and e is a prompt expression evaluating to Str . The semantics are: 1. The compiler generates a JSON Schema 

S(T ) from the struct definition of T at compile time. 2. At runtime, the VM suspends the pro-cess, constructs an inference request con-taining the prompt e, the agent’s context window, and S(T ), and delegates it to the configured Wasm inference driver. 3. The driver returns a JSON response. The VM validates the response against S(T ).4. If validation succeeds, the VM binds the parsed value to x with type T and re-sumes execution. 5. If validation fails, the VM re-prompts the model with the validation error (up to k

retries, default k = 3 ). If all retries fail, the VM raises a catchable error.

T { e } completes without error, then for every field fi : τi declared in struct T ,the access x.f i yields a value of type τi.

4.2 Typing Rule 

The T-Infer rule captures the hybrid static/-dynamic character of cognitive type safety: 

Γ ⊢ T : Struct S(T ) = s Γ ⊢ e : Str 

Γ ⊢ (let x = infer T {e}) : T (T-Infer) 

The first premise checks at compile time that 

T is a declared struct. The premise S(T ) = s

asserts that schema generation is total over struct types—it always succeeds, producing a concrete JSON Schema. The third checks that the prompt expression has type Str . The conclusion types x as T , which is sound by Theorem 4.1: the VM validates the model response against s before binding, so any suc-cessfully bound x satisfies the field-level type constraints of T .

# 5 Probabilistic Control Flow 

LLM inference is inherently stochastic. Even with Cognitive Type Safety guaranteeing struc-tural conformance, the semantic quality of a response may vary. Turn addresses this with the confidence operator. 

Definition 2 (Confidence Operator) . For any value v produced by an infer expression, the expression confidence v evaluates to a floating-point scalar c ∈ [0 , 1] representing the inference provider’s reported certainty. 

The confidence score is extracted by the Wasm inference driver from the model’s response metadata (e.g., top-token log probabilities for OpenAI with logprobs: true , or explicit confidence fields for providers that support them). For providers that expose neither, the driver returns a conservative default of 

0.5 —the neutral midpoint of the [0 , 1] range— preserving the invariant that confidence v 

always evaluates to a scalar in [0 , 1] . Programs that require a genuine certainty signal should select a provider that exposes log probabilities; programs that require only structural correct-ness can rely on Cognitive Type Safety inde-pendently of the confidence operator. The VM stores the score alongside the inferred value as an Uncertain<T> wrapper that is transparent to field access but queryable via confidence .The running example demonstrates this pat-tern: if the analyst’s thesis falls below 0.7 confidence, the program substitutes a deter-ministic “FAIR” verdict. The fallback branch executes natively in the VM with zero network latency and produces a value that structurally conforms to the same type. This enables a programming model where stochastic and de-terministic paths compose cleanly within the same type discipline. 

# 6 Actor-Based Agent Pro-cesses 

Turn’s concurrency model adapts Erlang’s ac-tor model [2] to the agentic domain. 

Definition 3 (Agent Process) . A Turn process is a 5-tuple (E, C, M, B, pc ) where: 

• E is the lexical environment (variable bindings), 

• C is the context window (ordered sequence of strings), 

• M is the persistent memory store (key-value map), 

• B is the mailbox (FIFO message queue), 

• pc is the program counter.

Processes are spawned with spawn ,

spawn_link , or spawn_each . Each re-ceives a fresh (E, C, M, B, pc ) tuple; no component is shared. 

6.1 Message Passing 

Inter-process communication uses asyn-chronous message passing: 

• send pid, value enqueues value in the mailbox of process pid . Non-blocking. 

• receive dequeues the next message from the current process’s mailbox. Blocks (yields to the scheduler) until a message is available. 

• self returns the current process’s Pid .

6.2 Fault Propagation 

spawn_link creates a bidirectional link between processes. When a linked process terminates, the VM generates a ProcessExit 

signal containing the exit reason and delivers it to all linked partners’ mailboxes. This enables Erlang-style supervisor trees: in the running example, the committee chair spawns two linked analysts; if either crashes, a 

ProcessExit signal is delivered to the chair’s mailbox, enabling deterministic fault recovery. 

6.3 Context Isolation and Struc-tured Memory 

A critical property of Turn’s actor model is 

context isolation : each process maintains its own context window. When an agent spawns a child, the child starts with an empty context. This prevents context contamination between agents, a common failure mode in framework-based multi-agent systems where a shared mes-sage list accumulates context from all agents indiscriminately. 

Property 1 (Context Isolation) . For any two processes Pi and Pj where i̸ = j, modifications to Ci have no effect on Cj .

Tripartite context architecture. Turn does not store context as a flat FIFO list. Each process context C is a three-tier 

StructuredContext :

• P0 (System). A Vec of system-level directives set by context.system(...) .Always rendered first in the flat context window, exploiting the primacy attention position ( ≈90% recall, [23]). 

• P1 (Working). A VecDeque of recent context items appended by 

context.append(...) , bounded at 

W = 100 entries. Always rendered last ,exploiting the recency position ( ≈85% recall, [23]). 

• P2 (Episodic). When P1 reaches capac-ity, the oldest working item is demoted to P2 rather than dropped; P2 is bounded at 2W = 200 entries. Rendered between 

P0 and P1, the middle zone of lower but non-zero recall. The flat rendering order P0 → P2 → P1 is en-forced by to_flat_vec() on every inference call. This design ensures that a Turn agent’s system role (P0) and most recent context (P1) always occupy the two attention-privileged po-sitions identified by Liu et al. [ 23 ], while older context is preserved in the episodic tier rather than silently truncated. See Experiment E3 for empirical verification of the eviction invariants and rendering order. 

6.4 Scatter/Gather: spawn_each 

Turn provides spawn_each as a native scat-ter/gather primitive for concurrent list pro-cessing. Each element is delegated to an in-dependent micro-actor with its own isolated context, memory, and mailbox. The expression 

spawn_each(list, turn(x: T) { ...}) is semantically equivalent to spawning one ac-tor per element and waiting for all to com-plete, but is a single language primitive rather than an application-level pattern. This re-places imperative loops with delegation, align-ing the programming model with the agentic paradigm where iteration is naturally parallel— see the investment committee example for a three-analyst case running concurrently under a spawn_link supervisor. 

# 7 Capability-Based Identity 

In agentic systems, the model controls the ex-ecution flow through its tool-calling decisions. 7Turn: A Language for Agentic Computation Preprint 

If API credentials are stored as strings in the program’s environment, a model that halluci-nates a call("echo", api_key) instruction can exfiltrate secrets. Turn prevents this with a capability-based identity system. 

Definition 4 (Identity Capability) . The expression grant identity:: σ(" name ") 

requests an opaque capability handle from the VM host, where σ is the capability class (e.g., 

oauth , network , filesystem ) and name 

identifies the provider. The VM returns a value of type Identity that: 1. Cannot be coerced to Str or any other type. 2. Cannot be serialized to JSON. 3. Cannot be passed to output tools (e.g., 

echo ). 4. Is enforced at two levels: the Standard Library validates the capability in pure Turn source code, and the kernel trap validates it in the Rust host. 

When an Identity handle reaches a kernel trap (e.g., __sys_http_get ), the VM host looks up the real credential from the process environment using the convention TURN_ IDENTITY_{NAME}_TOKEN and injects the 

Authorization header automatically. The raw token never enters Turn memory. The running example demonstrates both a network and a filesystem identity handle in use. 

# 8 Compile-Time Schema Ab-sorption 

Turn’s use schema::<protocol>("url") 

construct is a protocol-dispatching, network-aware compiler macro . The parser accepts any identifier as the protocol, and the schema com-piler dispatches to the corresponding adapter. The openapi adapter is fully implemented; 

graphql , fhir , and mcp adapters share the same pipeline and are in active development. 

8.1 Macro Expansion 

The schema compiler operates as a pre-analysis AST transformation pass: 1. The parser produces an Expr:: UseSchema AST node carrying the protocol and URL. 2. The schema compiler dispatches on the 

protocol field and fetches the API speci-fication from the URL at compile time. 3. For each endpoint or type definition, it generates: (a) a Turn struct corresponding to the response schema, and (b) a closure that constructs the appropriate request. 4. The UseSchema node is replaced with an immediately invoked function expression (IIFE) containing the generated structs and closures. 5. The transformed AST proceeds to semantic analysis and compilation. The UseSchema node never reaches the bytecode emitter. This enables static verification that infer out-put (typed by a synthesized struct) structurally conforms to the API’s expected input format, closing the type gap between LLM output and external API consumption. 

End-to-end chain. A program writes use schema::openapi(...) with a Stripe API URL. The schema compiler fetches the spec, synthesizes a Customer struct and a typed clo-sure. At runtime, an Identity handle scopes the HTTP call; the programmer never writes a JSON Schema. A subsequent infer expres-sion produces a value whose fields are statically guaranteed by Theorem 4.1. A full worked ex-ample is in the repository.

Turn’s P0/P1/P2 tripartite context enforces the primacy-recency rendering invari-ant: system prompts always occupy the pri-macy position and the most recently appended item always occupies the recency position, re-gardless of context volume. Working memory overflow demotes to episodic rather than drop-ping. 

Background. Liu et al. [ 23 ] show recall follows a U-shaped curve: ≈90% for primacy-position content, ≈50% for middle, ≈85% for recency. Every framework-based system that appends messages to a flat FIFO list loses this advantage as the list grows and the model truncates from one end. Turn’s 

StructuredContext solves this structurally, not by convention. 

Mechanism. to_flat_vec() always emits in the order p0_system → p2_episodic →

p1_working . P1 is a bounded VecDeque (W =100 ); when full, the oldest working item is moved to P2 (bounded at 2W = 200 ). The primacy and recency positions are thus held by P0 and the tail of P1 regardless of total context size. 10 Turn: A Language for Agentic Computation Preprint 

Results. Five tests verify the invariants (all pass on Apple M4): 

• E3-1 : P0 system prompt is always index 0 in the flat context vector. 

• E3-2 : Most recently appended P1 item is always the last element. 

• E3-3 : At item 101, P1 size remains at 100; exactly one item is evicted to P2 (demote, not drop). 

• E3-4 : Flat rendering confirmed as P0(idx 0) → P2(idx 1) → P1(idx 2–101). 

• E3-5 : Each spawned process starts with an empty context (isolation invariant).

10.6 Architectural Complexity vs. Frameworks 

While Turn reduces the raw lines of code (LOC) required to implement the investment committee from an estimated 350+ lines in LangChain to 89 lines, the more significant reduction is in state space complexity . In a Python framework, achieving fault-tolerant concurrency, durable agent memory, and structured output requires an external Redis instance for state, a message queue (e.g., Celery) for worker processes, and explicit retry loops around JSON parsers. In Turn, these are collapsed into the language runtime: 

spawn_link handles concurrency and fault propagation without external queues, suspend 

orthogonalizes persistence without database boilerplate, and infer absorbs the retry loop. Turn reduces a distributed systems architecture into a single-process actor topology.

# 11 Discussion 

11.1 Limitations 

Turn is a domain-specific language for agentic orchestration, not a general-purpose systems language. It intentionally omits low-level mem-ory management and GPU computation, dele-gating heavy workloads to WebAssembly com-ponents or microservices via call . However, Turn is architected to scale: its spawn_link 

supervisor trees support massive multi-agent platforms—such as our internal deployment of the JarvisCore autonomous system—where complexity lies in state persistence and capa-bility routing rather than raw computational throughput. While infer guarantees structural shapes, standard variable bindings currently remain dy-namically typed, meaning non-inference type errors are caught at runtime. Future iterations may introduce optional static typing. Addi-tionally, while developers can build robust re-covery loops using Turn’s native try /catch 

constructs, integrating LLM self-reflection [ 27 ]directly into the language syntax remains an open direction. Finally, the confidence operator relies on the underlying inference provider exposing log probabilities. For opaque models, the VM must fall back to default scores, limiting the utility of probabilistic routing. 12 Turn: A Language for Agentic Computation Preprint 

11.2 Future Work 

The current implementation of Turn estab-lishes the core primitives for language-oriented agentic development. However, realizing the full vision of a production-grade ecosystem requires substantial ongoing work in several directions: 

• Extended Schema Adapters : Expand-ing use schema::<protocol> to support 

graphql , fhir , mcp , and odata , covering the full landscape of enterprise and health-care APIs. 

• Distributed Actor Scheduling : Ele-vating the Erlang-inspired concurrency model to support distributed actor scheduling across multiple nodes via the existing gRPC switchboard. 

• Tooling and Ecosystem : Developing a robust Language Server Protocol (LSP) implementation for advanced IDE sup-port, and a native package manager for distributing Turn modules and schema adapters. 

• Formal Verification : Extending the for-mal semantics of Cognitive Type Safety to full mechanical verification of the com-piler’s schema generation and the VM’s validation pipeline. 

• Multi-modal Primitives : Extending the infer primitive to natively accept im-age, audio, and video inputs, and explor-ing broader probabilistic reasoning con-structs. 

Call for Collaboration. Turn is an ambi-tious undertaking that sits at the intersection of programming language theory and artificial intelligence. We actively invite researchers, compiler engineers, and open-source contribu-tors to collaborate on these future directions and help shape the foundation of agentic soft-ware. 

# 12 Conclusion 

We have presented Turn, a programming lan-guage that makes the inherent structure of agentic computation explicit and enforceable. By introducing Cognitive Type Safety, prob-abilistic control flow, actor-isolated processes, capability-based identity, and compile-time schema absorption as first-class language con-structs, Turn eliminates five fundamental fail-ure modes of framework-based agent develop-ment. 

> Key Insight

An agent is a process, not a loop. It has bounded context, persistent memory, a mailbox, and a lifecycle. Encoding these constraints in the language lets the com-piler and runtime enforce them mechani-cally, eliminating the entire class of fail-ures that arise when critical invariants are left to developer discipline. Turn demonstrates that a small, focused language with the right primitives provides stronger guarantees for agentic software than any library built on a general-purpose host.
