## 摘要
多智能体系统（MAS）应如何设计？这些设计知识能否以一种可检查、可修订且可迁移的形式被捕获？我们提出了 ABSTRAL，这是一个将 MAS 架构视为不断演进的自然语言文档的框架，该文档是通过对比轨迹分析（contrastive trace analysis）进行细化的产物。研究得出了三项发现：首先，我们对多智能体协调税（multi-agent coordination tax）进行了精确测量：在固定轮次预算下，集成系统仅达到 26% 的轮次效率，66% 的任务耗尽了限制，但通过发现可并行的任务分解，其表现仍优于单智能体基准。其次，编码在文档中的设计知识具有可迁移性：在一个领域学习到的拓扑推理和角色模板为新领域提供了先发优势，迁移后的种子在单次迭代中即可达到冷启动第 3 次迭代的性能。第三，对比轨迹分析能够发现初始设计中不存在的专家角色，这是以往系统所不具备的能力。在 SOPBench（包含 134 个银行任务，确定性预言机）上，以 GPT-4o 为骨干网络的 ABSTRAL 达到了 70% 的验证集通过率和 65.96% 的测试集通过率。我们发布了收敛后的文档，作为可检查的设计原理。

## Abstract
How should multi-agent systems be designed, and can that design knowledge be captured in a form that is inspectable, revisable, and transferable? We introduce ABSTRAL, a framework that treats MAS architecture as an evolving natural-language document, an artifact refined through contrastive trace analysis. Three findings emerge. First, we provide a precise measurement of the multi-agent coordination tax: under fixed turn budgets, ensembles achieve only 26% turn efficiency, with 66% of tasks exhausting the limit, yet still improve over single-agent baselines by discovering parallelizable task decompositions. Second, design knowledge encoded in documents transfers: topology reasoning and role templates learned on one domain provide a head start on new domains, with transferred seeds matching coldstart iteration 3 performance in a single iteration. Third, contrastive trace analysis discovers specialist roles absent from any initial design, a capability no prior system demonstrates. On SOPBench (134 bank tasks, deterministic oracle), ABSTRAL reaches 70% validation / 65.96% test pass rate with a GPT-4o backbone. We release the converged documents as inspectable design rationale.

#### 1. 核心问题与整体含义（研究动机和背景）
当前的多智能体系统（MAS）大多依赖人工设计，现有的自动化方法通常将中间知识存储为代码、提示词或技能库，缺乏一种**可检查、可修订且可迁移**的“设计知识”表达形式。论文探讨了自然语言文档是否可以作为 MAS 设计的优化目标，旨在解决如何自动构建高效拓扑结构、发现缺失角色以及在不同任务领域间迁移设计经验的问题。

#### 2. 方法论：核心思想与算法流程
ABSTRAL 框架将 MAS 架构视为一个动态演化的 Markdown 文档（`SKILL.md`），通过三层循环进行优化：
*   **核心数据结构 (`SKILL.md`)**：包含四个部分：**K**（领域知识）、**R**（拓扑推理规则）、**T**（发现的角色模板）和 **P**（构建协议）。
*   **第一层：内部循环（轨迹驱动细化）**：
    1.  **构建 (Build)**：元智能体根据文档生成智能体规范并实例化系统。
    2.  **运行 (Run)**：在沙盒中执行任务并捕获执行轨迹。
    3.  **分析 (Analyze)**：对比成功与失败的轨迹，将失败归类为五种证据类别（EC1-EC5，如拓扑瓶颈、角色缺失等）。
    4.  **更新 (Update)**：根据证据对文档特定部分进行针对性编辑。
*   **第二层：收敛与整合**：监控性能平台期和规则冗余，定期进行“整合”以消除语义漂移，压缩冗余规则。
*   **第三层：外部循环（拓扑多样性）**：利用**图编辑距离 (GED)** 和语义嵌入距离作为排斥约束，强制系统探索性质完全不同的拓扑家族（如从层级结构转向集群结构），防止陷入局部最优。

#### 3. 实验设计
*   **数据集/场景**：使用 **SOPBench**（标准作业程序基准测试）中的银行领域任务，包含 134 个任务。
*   **评测标准**：采用确定性预言机（Oracle）评估，必须满足 5 项布尔准则（无工具调用错误、无约束违反、数据库匹配等）才算通过。
*   **对比方法**：
    *   **单智能体基准**：GPT-4o, Claude-3.7-Sonnet, o4-mini-high 等。
    *   **消融实验**：仅内部循环（固定拓扑）vs. 全框架（带外部拓扑搜索）；有无知识迁移的对比。

#### 4. 资源与算力
*   **模型使用**：元智能体使用 Claude Sonnet 4；执行智能体骨干网络使用 GPT-4o。
*   **算力/成本**：
    *   **Token 消耗**：整个搜索过程消耗约 15.4 万个 GPT-4o token。
    *   **经济成本**：API 费用约 5 美元（骨干网络）+ 0.5 美元（元智能体）。
    *   **时间成本**：总耗时 28.4 小时，其中 79% 的时间用于任务执行（RUN 阶段）。
*   **硬件环境**：沙盒环境配置为 2GB RAM。

#### 5. 实验数量与充分性
*   **实验规模**：进行了 3 个外部循环，每个循环包含 8 次内部迭代，共 24 次完整迭代。
*   **充分性**：实验涵盖了从单智能体到多智能体、从层级到集群拓扑的演进过程。通过消融实验验证了外部循环对性能提升的贡献（+15%）。
*   **客观性**：使用了持有测试集（94 个任务）进行最终评估，避免了过拟合验证集的问题。但论文也坦诚指出，由于样本量和随机性，不同运行之间存在约 5% 的性能波动。

#### 6. 主要结论与发现
*   **协调税（Coordination Tax）的量化**：首次精确测量了多智能体开销。在固定轮次预算下，多智能体系统仅有 26% 的轮次用于实际工具调用，66% 的任务会触及轮次上限。
*   **设计知识的可迁移性**：在一个领域学习到的 `SKILL.md` 文档可以显著加速新领域的优化。迁移后的种子在 1 次迭代内即可达到冷启动 3 次迭代的水平。
*   **角色自动发现**：通过对比轨迹分析（EC3），系统能自动识别出需要专门角色的场景（如发现需要专门的“外汇专家”或“验证者”），并生成相应的系统提示词。
*   **性能表现**：在 SOPBench 银行任务上，ABSTRAL 达到了 65.96% 的测试通过率，优于原始 GPT-4o 单智能体表现。

#### 7. 优点
*   **可解释性强**：优化目标是自然语言文档，人类可以直接阅读并理解系统为何选择某种拓扑或角色。
*   **拓扑感知优化**：不同于以往只优化 Prompt 的系统，ABSTRAL 能在不同的图结构家族间进行搜索。
*   **发现新角色**：具备从执行失败中推断并创建初始设计中未定义的专家角色的能力。

#### 8. 不足与局限
*   **轮次效率瓶颈**：由于多智能体间的路由和通信消耗了大量轮次，在严苛的轮次限制下，多智能体可能反而不如单智能体（尤其在长链顺序任务中）。
*   **分类体系限制**：拓扑分类目前仅限于 7 种功能类型，可能无法覆盖所有潜在的架构。
*   **框架开销**：实验中使用的 LangGraph 框架本身在节点转换时会消耗轮次，这与直接进行函数调用的基准测试之间存在一定的对比偏差。
